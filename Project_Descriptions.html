    • Study this sequential implementation of word count, https://gitlab.com/SpiRITlab/CoPaDS/tree/master/src/main/java/edu/rit/cs/basic_word_count
    • Learn about the use of OO programming paradigm for modeling the AmazonFineFoodReview

    • Measure the time it takes

  • Sub-task 2:Speed up this program using multiple threads (5%)

    • Sort each partition independently (sort by words in ascending order)

    • Measure the time it takes
    • Create a new class file name, e.g., WordCount_Threads

 

  • Sub-task 3: Speed up this program using multiple nodes and multiple threads on each node (5%)

    • Merge and sort all results.
    • Measure the time it takes
    • Create two new classes with file names, e.g., WordCount_Cluster_master.java and WordCount_Cluster_worker.java

 

 

 

 

 

 

 

 

Project 2: (Group) Implement a Publish/Subscribe system (10%)

Objective: Provide students hand-on experience with pub/sub messaging system.

 

  • Extend the provided classes for modeling a pub/sub system, topics, and events.
  • Create appropriate data structure for storing topics, events, and user info
  • Implement the five key operations (subscribe, unsubscribe, publish, advertise, notify)
  • Implement functions to display the current list of subscribers on the event manager (need multi-threading in terminal)
  • Provide necessary command line interface to support key operations; for example, describing event of interest
  • Support asynchronous event notification: when a subscriber is offline, the system caches events intended for this subscriber and send to the subscriber when reconnected.
  • Testing procedures for the basic Pub/Sub functionalities (3% for each and 1% for documentation): 
    • A publisher advertises a new topic. This new topic will be broadcasted to all nodes (both publishers and subscribers). 
    • A subscriber subscribes to a topic. When a new event is published, only the subscribed subscribers will receive the event.
    • A subscriber goes offline, during this time a publisher publishes a new event. The event manager will cache the event. When the subscriber comes online, the event is sent to this subscriber. 

 

 

 

 

 

Project 3: (Group) Implement Chord distributed hash table (10%)

Objective: Provides students hand-on experience with peer-to-peer systems.

 

  • Model elements that you will need in developing Chord, for example, object classes for providing structured format for data object.
  • Create a network that can supports upto n=24=16 nodes (Docker containers) as an example.
    • In production system, it is often n=2256, so your code should not be hard coded.
  • Implement a Chord DHT using these nodes
  • Create files with the hash code, hash(filename+filesize) mod n, and insert these files from a node. The placement of files should be according to how Chord works.
  • Write code to address fault-tolerance when nodes join and leave; that is, moving files around
  • Demonstrate the ability to perform query of the files from any node
  • Testing procedures for the Chord functionalities (1% for documentation):
    • With 5 random nodes on the network, make sure the finger tables are updated correctly (3%).
      • For simplicity, we will use the first node (ID=1) as an anchor node (active all the time).
      • Bonus points (1 pt): what can we do to remove this anchor node? You can propose an idea to the instructor and implement the solution to claim the bonus points.
    • Place a file into the network from any active node and query the file (2%)
    • Add a new node with a random ID and check the finger table and file placement (2%)
    • Remove an existing node with a random ID and check the finger table and file placement (2%)

 

 

Bonus points are added to compensate any point that is lost due to other issues in your solution.
